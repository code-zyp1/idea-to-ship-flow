# 想法到交付三步流（Idea → Spec → Ship）

一套适合「需求很模糊 / 你不熟的领域 / 只想尽快开工」的轻量工作流模板。

核心目标：把零散想法快速收敛成**可执行规格**，再自动生成**可提交的工程文档 + 启动提示词**，直接进入编码与审计。

---

## 你会得到什么

- **想法孵化**：当你只有一句话/几张截图时，用最少轮次把问题变成“选择题”，收敛出可执行方向  
- **澄清规格（Spec Brief）**：生成一份可开工的规格，包含验收标准、范围锁定、降级策略、并行拆分、工程约束  
- **工程文档生成器**：把 Spec Brief 一键变成：
  - `docs/prd.md`
  - `docs/plan.md`
  - `docs/decisions.md`
  - `docs/checklist.md`
  - + Claude Code 实现启动提示词
  - + Codex/GPT 审计启动提示词

---

## 文件说明

- `想法孵化.md`
  - 适合一开始信息极少的场景（只有一句话/很乱的描述）
  - 输出短、可多轮对话推进，不会一上来写长 PRD

- `澄清规格.md`
  - 把讨论收敛为《Spec Brief》
  - 默认按“v1 就要达到最终效果”的倾向编写，同时提供范围锁定与降级策略

- `工程文档生成器.md`
  - 输入 Spec Brief
  - 输出可直接提交到仓库的文档 + 两段启动提示词（实现 / 审计）
  - 会把工程约束/skills（如有）注入到 plan 中

---

## 使用方式（推荐：网页端 Chatbox）

### Step 0：只有一句话/很模糊
把你的想法、截图要素、参考链接等内容粘贴给模型，并附上 `想法孵化.md` 的提示词。

当模型回复里出现：
> ✅ 已可进入：澄清规格（Spec Brief）

就进入下一步。

### Step 1：生成 Spec Brief
把「想法孵化的结论 + 你的补充」粘贴给模型，并附上 `澄清规格.md` 的提示词，拿到完整 Spec Brief。

建议你只检查三件事（1 分钟）：
- 目标一句话是否正确
- 必须保留/不能发生是否符合预期
- 验收标准是否可测试、可判定

### Step 2：生成工程文档 + 启动提示词
把 Spec Brief 全文粘贴给模型，并附上 `工程文档生成器.md` 的提示词。

你会得到：
- 4 份工程文档（可直接提交到仓库）
- 2 段启动提示词：
  - 给 Claude Code：实现开工指令
  - 给 Codex/GPT：审计 review 指令

## 适用场景

特别适合：
- 截图/竞品复刻（小程序/网页 UI）
- 从 0 到 1 的小中型工具/产品原型
- 你不熟的领域，需要“默认假设 + 可验证验收”来推进
- 想减少反复沟通，直接让模型产出可开工的结构化规格

不太适合：
- 纯研究/论文型（交付不是可运行系统）
- 已经有非常严格外部规格/接口验收表的项目（可用但会显得偏轻）

---

## 设计原则

- **默认一次到位倾向**：v1 以最终效果为目标  
- **范围锁定 + 降级策略**：防止 scope 膨胀，也避免卡住无法交付  
- **工程化验收**：能跑、像、可用、稳、数据可替换  
- **并行友好**：impl/ui/test 的 Ownership 与契约，减少冲突  
- **输出可落库**：文档不依赖对话上下文，可直接提交
